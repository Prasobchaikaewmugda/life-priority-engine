## 0. บทสรุปผู้บริหาร

โปรเจกต์นี้ไม่ควรเป็นแค่ “เว็บวางแผนรายวัน” หรือ “todo planner” แต่ควรเป็น **ผู้ช่วยจัดลำดับชีวิตส่วนตัว** ที่รู้บริบทชีวิตจริงก่อน แล้วค่อยตอบว่า:

- วันนี้ควรทำอะไร
- ทำไมควรทำสิ่งนั้นก่อน
- ควรลด/เลื่อน/ตัดอะไร
- ควรพักตอนไหน
- ควรอ่านหนังสือตอนไหน
- ควรออกกำลังกายแบบไหน
- ควรกินอะไรแบบง่าย ๆ
- ควรหยุดงานหนักกี่โมงเพื่อไม่กระทบพรุ่งนี้
- แผนวันนี้ควรปรับจากผลลัพธ์เมื่อวานอย่างไร

แกนผลิตภัณฑ์ที่ถูกต้องคือ:

```text
Core Life Profile
+ Daily Context
+ Daily Plan
+ Task Result
+ Daily Reflection
+ Learned Patterns
+ Persistence
```

จุดแข็งที่สุดของโปรเจกต์คือ **ตั้งค่าชีวิต + เช็คอินวันนี้ + เรียนรู้จากผลจริง** ไม่ใช่แค่สร้างภารกิจทั่วไป

## 1. Product Philosophy

### 1.1 เป้าหมายแท้จริง

> ช่วยผู้ใช้ตัดสินใจว่า “ตอนนี้ควรทำอะไร” โดยอิงจากชีวิตจริง พลังงาน เวลา งาน สุขภาพ เป้าหมาย และสิ่งที่เกิดขึ้นในแต่ละวัน

ไม่ใช่:

```text
- สร้าง todo list ทั่วไป
- ให้คำแนะนำ generic
- บอกให้อ่านหนังสือ/ออกกำลังกาย/พัก โดยไม่รู้บริบท
- ให้ผู้ใช้กรอกข้อมูลเยอะแล้วไม่สรุปอะไรกลับ
```

### 1.2 ผู้ใช้ต้นแบบ

ผู้ใช้เริ่มต้นคือเจ้าของโปรเจกต์เองก่อน:

```text
- ทำงานเป็นเวร
- มีเป้าหมายอ่านหนังสือ
- มีโปรเจกต์ที่อยากทำ
- อยากดูแลสุขภาพ
- มีปัญหานอนดึก / เหนื่อยหลังเลิกงาน / ไม่รู้ควรทำอะไรก่อน
```

ถ้าแอปแก้ปัญหาให้เจ้าของได้จริง จึงค่อยขยายไปผู้ใช้คนอื่น

### 1.3 หลักการออกแบบ

```text
1. ถามน้อยเท่าที่จำเป็น แต่เปิดให้เล่าละเอียดได้
2. ทุกคำตอบต้องสรุปกลับว่า “ระบบเข้าใจอะไร”
3. ทุกแผนต้องบอกเหตุผล
4. ทุกแผนต้องมี fallback ถ้าเหนื่อย
5. ข้อมูลรายวันไม่ควรเขียนทับข้อมูลชีวิตโดยรวมทันที
6. ถ้าข้อมูลไม่พอ ต้องบอกว่าไม่พอ ไม่เดามั่ว
7. สุขภาพ/อาหาร/เงิน ต้องเป็นคำแนะนำทั่วไป ไม่ใช่คำแนะนำแพทย์หรือการเงินเฉพาะบุคคล
8. ทำทีละ phase พร้อม PASS/HOLD gate
```

## 2. วิเคราะห์จุดดี จุดด้อย และแนวทางที่ดีกว่า

### 2.1 จุดดีของแนวคิดปัจจุบัน

```text
1. ตั้งค่าชีวิตเป็นหัวใจของระบบ
2. ให้ผู้ใช้เล่าเรื่องชีวิต ไม่ใช่ติ๊ก checkbox อย่างเดียว
3. ต้องการให้เว็บวิเคราะห์ละเอียดและตรงตัวผู้ใช้
4. มีแนวคิดบันทึกผลหลังทำจริง
5. ต้องการให้วันถัดไปปรับตามสิ่งที่เกิดขึ้นจริง
6. เห็นชัดว่า planner ทั่วไปไม่พอ ต้องมีบริบทชีวิต
```

### 2.2 จุดด้อย / ความเสี่ยง

```text
1. ถ้าเอาทุกข้อมูลไปรวมในหน้าเดียว จะสับสนว่าอะไรเป็นข้อมูลถาวรและอะไรเป็นข้อมูลเฉพาะวัน
2. ถ้าใช้แต่ช่องเล่ายาว ๆ โดยไม่มีระบบสรุป จะกลายเป็น “พิมพ์แล้วหาย”
3. ถ้าไม่มี persistence ข้อมูลจะหายและผู้ใช้จะไม่อยากใช้ต่อ
4. ถ้าไม่มี reflection loop ระบบจะไม่เรียนรู้
5. ถ้าไม่มี learned patterns ระบบอาจสรุปผิดจากเหตุการณ์วันเดียว
6. ถ้ารวมสุขภาพ/เงินลึกเกินไป อาจ scope ใหญ่เกินและเสี่ยง high-stakes
7. ถ้าไม่มี confidence/missing information ระบบจะดูเหมือนมั่วเมื่อข้อมูลไม่ครบ
```

### 2.3 แนวทางที่ดีกว่า

ใช้ 3 ชั้นหลัก:

```text
ชั้น 1: Core Life Profile
ข้อมูลชีวิตโดยรวมที่ค่อนข้างคงที่

ชั้น 2: Daily Context
ข้อมูลเฉพาะวันนี้ เช่น นอนน้อย เวรหนัก ฝนตก เครียด มีงานด่วน

ชั้น 3: Reflection + Learned Patterns
สิ่งที่เกิดขึ้นจริงหลังใช้แผน และบทเรียนระยะยาว
```

เปลี่ยนแนวคิดหลัก:

```text
จาก: ตั้งค่าชีวิต = กรอกข้อมูลทั้งหมด
เป็น: ตั้งค่าชีวิต = ฐานชีวิตระยะยาว 5 โหมด

จาก: ภารกิจรายวัน = สร้างจาก template
เป็น: ภารกิจรายวัน = สร้างจาก profile + today context + reflection

จาก: บันทึกผล = แค่ติ๊ก
เป็น: บันทึกผล = feedback loop เพื่อปรับวันถัดไป
```

## 5. Core Life Profile — ตั้งค่าชีวิต 5 โหมด

หน้า “ตั้งค่าชีวิต” ควรเป็น 5 โหมดแบบเล่าเรื่อง:

```text
1. เป้าหมายและปัญหาที่อยากแก้
2. เวลาและจังหวะชีวิต
3. สุขภาพและพลังงาน
4. งาน / เงิน / ภาระที่ห้ามพลาด
5. เรื่องราวชีวิตและกฎส่วนตัว
```

แต่ละโหมดต้องมี 4 ชั้น:

```text
1. ช่องเล่าเรื่องละเอียด
2. คำถามนำทาง
3. สรุปสิ่งที่ระบบเข้าใจ
4. ระบบจะใช้ข้อมูลนี้เพื่ออะไร
```

### 5.1 โหมด 1 — เป้าหมายและปัญหาที่อยากแก้

#### ข้อมูลที่ควรเก็บ

```text
- เป้าหมายหลัก
- เป้าหมายรอง
- deadline
- เหตุผลที่เป้าหมายสำคัญ
- ปัญหาที่ทำให้ไม่สำเร็จ
- ถ้าวันนี้ทำได้อย่างเดียว ควรทำอะไร
- สิ่งที่ห้ามกระทบ เช่น สุขภาพ งานประจำ การนอน
```

#### จุดดี

```text
- เริ่มจาก pain point ของผู้ใช้
- ทำให้ผู้ใช้รู้สึกว่าเว็บกำลังช่วยแก้ปัญหา
- ใช้จัด priority ได้ชัด
```

#### จุดด้อย

```text
- ผู้ใช้มักมีหลายเป้าหมายพร้อมกัน
- ถ้าไม่บังคับ priority ระบบจะสับสน
- ถ้าเป้าหมายกว้าง เช่น “อยากดีขึ้น” จะวิเคราะห์ยาก
```

#### ควรลด

```text
- อย่าให้ทุกเป้าหมายสำคัญเท่ากัน
- อย่าถามเป้าหมายยาวมากโดยไม่มี deadline หรือ minimum action
```

#### ควรเพิ่ม

```text
- primary_goal
- secondary_goals
- goal_type
- urgency
- failure_pattern
- minimum_action_today
```

#### ตัวอย่าง summary

```text
ระบบเข้าใจว่า:
- เป้าหมายหลักคืออ่านหนังสือให้ทันสอบ
- เป้าหมายรองคือสุขภาพและโปรเจกต์
- จุดเสี่ยงคือเหนื่อยหลังเลิกเวรและนอนดึก
- ถ้าวันนี้พลังต่ำ ระบบควรให้ทำขั้นต่ำคืออ่าน 20–30 นาที
```

### 5.2 โหมด 2 — เวลาและจังหวะชีวิต

#### ข้อมูลที่ควรเก็บ

```text
- เวลาตื่นปกติ
- เวลานอนเป้าหมาย
- เวลาเริ่มงาน
- เวลาเลิกงาน
- เวลาเดินทาง
- ช่วงสมองดีที่สุด
- ช่วงพลังตก
- เวลาที่ไม่ควรเริ่มงานหนัก
- กฎก่อนวันเวรเช้า
```

#### จุดดี

```text
- เป็นหัวใจของคำถาม “เวลานี้ควรทำอะไร”
- ทำให้แผนไม่ชนงาน ไม่ชนการนอน
- ทำให้ timetable มีเหตุผล
```

#### จุดด้อย

```text
- ถ้าถามเวลาละเอียดเกิน ผู้ใช้จะเหนื่อย
- ถ้าถามน้อยเกิน แผนยัง generic
```

#### ควรลด

```text
- ไม่ต้องเริ่มจากตารางทั้งสัปดาห์
- ไม่ต้องถามเวลาทุกกิจกรรมตั้งแต่แรก
```

#### ควรเพิ่ม

```text
- daily_free_window
- no_heavy_work_after
- best_focus_time
- low_energy_time
- shift_specific_rule
```

#### ตัวอย่าง summary

```text
ระบบเข้าใจว่า:
- เวลาว่างจริงหลังงานคือ 17:00–22:30
- 17:30 เหมาะกับเดินเบา
- 19:00–20:30 เหมาะกับอ่านหนังสือ
- หลัง 21:30 ไม่ควรเปิดงาน deep work ใหม่
```

### 5.3 โหมด 3 — สุขภาพและพลังงาน

#### ข้อมูลที่ควรเก็บ

```text
- เรื่องสุขภาพโดยรวม
- ส่วนสูง / น้ำหนัก เฉพาะถ้าผู้ใช้ต้องการใช้เป้าหมายสุขภาพ
- เป้าหมายสุขภาพ เช่น ลดน้ำหนัก เพิ่มน้ำหนัก คุมสุขภาพ ฟื้นพลัง
- พลังงานหลังเลิกงาน
- การนอน
- ออกกำลังกายที่ชอบ
- ข้อจำกัดร่างกาย
- อาหารที่กินง่าย
- อาหารที่ไม่กิน
```

#### จุดดี

```text
- ทำให้แผนไม่ฝืนร่างกาย
- แยกวันพลังดี/พลังต่ำได้
- ใช้ปรับออกกำลังกายและอาหารให้เหมาะกับวัน
```

#### จุดด้อย

```text
- เสี่ยงกลายเป็นคำแนะนำแพทย์ถ้าเขียนไม่ระวัง
- BMI หรือแคลอรี่ละเอียดอาจทำให้ scope ใหญ่เกิน
```

#### ควรลด

```text
- ไม่ต้องเริ่มจาก macro ละเอียด
- ไม่ต้องคำนวณ TDEE แบบจริงจังใน MVP
- ไม่ควรบอกแผนอาหารเหมือนแพทย์
```

#### ควรเพิ่ม

```text
- sleep_hours
- energy_level
- pain_or_discomfort
- exercise_intensity_limit
- simple_food_options
- recovery_rule
```

#### ตัวอย่าง summary

```text
ระบบเข้าใจว่า:
- วันทำงานควรออกกำลังเบา 15–30 นาที
- ถ้านอนน้อยกว่า 6 ชม. ไม่ควรวางวิ่งหนัก
- หลังออกกำลังควรเลือกอาหารโปรตีนง่าย เช่น ไข่ อกไก่ นม
```

### 5.4 โหมด 4 — งาน / เงิน / ภาระที่ห้ามพลาด

#### ข้อมูลที่ควรเก็บ

```text
- งานประจำ / เวร
- งานที่ห้ามพลาด
- รายได้หลักหรือสิ่งที่กระทบชีวิต
- โปรเจกต์ที่มีผลต่ออนาคต
- ภาระครอบครัวหรือความรับผิดชอบ
- ถ้าวันนี้งานหนัก ระบบควรลดอะไร
```

#### จุดดี

```text
- กันไม่ให้ระบบแนะนำสิ่งที่กระทบงานหรือรายได้
- ช่วยจัด priority เมื่องาน/เรียน/สุขภาพชนกัน
```

#### จุดด้อย

```text
- ถ้าทำเรื่องเงินลึก จะหลุดไปเป็น finance app
- อาจเพิ่มความกังวลให้ผู้ใช้ถ้าถามเรื่องเงินละเอียดเกิน
```

#### ควรลด

```text
- ยังไม่ทำบัญชีรายรับรายจ่าย
- ยังไม่ทำคำแนะนำลงทุน
- ยังไม่ทำวิเคราะห์หนี้สินลึก
```

#### ควรเพิ่ม

```text
- must_not_miss
- income_protection_rule
- work_fatigue_rule
- project_income_potential
- priority_conflict_rule
```

#### ตัวอย่าง summary


## 16. คำตัดสินสุดท้าย

ทิศทางที่ดีที่สุดคือ:

```text
ตั้งค่าชีวิต = รู้จักผู้ใช้ระยะยาว
เช็คอินวันนี้ = รู้สภาพจริงวันนี้
แผนวันนี้ = ตัดสินใจจากข้อมูลทั้งสอง
บันทึกผล = รู้ว่าแผนเวิร์กไหม
เรียนรู้ = ใช้หลายวันปรับให้แม่นขึ้น
```

นี่คือสิ่งที่จะทำให้โปรเจกต์มีจุดแข็งจริง:

```text
ไม่ใช่แค่ “วันนี้ทำอะไร”
แต่คือ “ทำไมวันนี้ควรทำสิ่งนี้ และควรลดอะไรเพื่อไม่พังพรุ่งนี้”
```
